## 一、提案背景

目前 iFare 後台已具備完整的內容維護功能，涵蓋最新消息、福利專欄、懶人包、福利政策、FAQ、洽辦單位、公益夥伴、代碼維護、圖片管理、帳戶管理與資料分析等模組。

從現有系統結構與程式碼設計來看，後台已建立出明確的 CRUD 管理架構，具備一定的穩定度與可延伸性。然而，隨著功能模組、欄位數量與維護需求逐漸增加，系統也進入需要進一步整理與標準化的階段。

目前後台的主要問題並非功能不足，而是集中在以下幾個面向：

1. 模組之間操作體驗可能不夠一致
2. 大型表單維護成本高
3. 權限邏輯與 API 結構維護性有待提升
4. 共用元件與錯誤回饋尚可再系統化

因此，本提案希望在**不推翻既有架構**的前提下，針對後台進行分階段優化，以提升操作效率、降低錯誤率，並改善後續維護性。

## 二、提案目標

本次後台優化方案的目標如下：

### 1. 提升後台操作效率
讓管理者在查詢、編輯、維護資料時更順手，減少不必要的操作成本。

### 2. 降低使用與維護成本
透過操作流程優化、表單重整與結構標準化，降低使用者與開發者的負擔。

### 3. 改善系統可維護性
強化共用結構、整理 API 與權限邏輯，使後續擴充與維護更容易。

### 4. 提升內部管理品質
使後台不只是可用，而是更穩定、更清楚、可持續成長。

## 三、現況問題盤點

### 3.1 CRUD 架構完整，但模組間重複度高
目前多數模組皆採用以下頁型結構：

- DataList
- AddEdit
- Detail
- Index

此架構代表後台已有一致的開發方向，但如果缺少更高層級的抽象與共用化，容易造成以下問題：

- 相似頁面分散維護
- 新增模組時需重複開發
- 改版時需同步修改多處

### 3.2 列表頁查詢流程可能不夠一致
從部分列表頁結構可看出，目前查詢流程多依賴：

- 查詢條件元件
- watch 條件變化
- API 重新抓取資料

這種方式雖然可運作，但如果各模組未統一規格，可能出現：

- 查詢送出時機不同
- 重設條件行為不一致
- 空狀態與錯誤狀態不清楚
- 使用者不易判斷是否查詢成功

### 3.3 福利政策維護表單資訊量大，操作負擔高
福利政策維護頁面屬於高複雜度表單，包含：

- 政策類別
- 地區
- 受助者
- 經濟條件
- 特殊身分
- 關鍵字
- 標題
- 申請資格
- 福利內容
- 應備證件
- 洽辦單位
- 上下架日期
- 備註

此類大型表單若未經優化，常見問題包括：

- 欄位多，填寫負擔高
- 使用者容易漏填或誤填
- 修改時不易快速掌握欄位位置
- 儲存失敗時處理成本高

### 3.4 權限控管邏輯可維護性不足
目前路由權限主要透過程式中的條件判斷進行控制，雖然可以運作，但存在以下風險：

- 新增頁面時容易漏改
- 權限規則不夠集中
- 難以快速對照角色與頁面關係
- 不利於後續文件化與交接

### 3.5 API 呼叫層過度集中
目前後台 API 呼叫邏輯集中於單一封裝檔案中，短期有利於快速開發，但隨著系統模組增加，容易出現：

- 檔案過大
- 模組責任邊界不清楚
- 維護與除錯難度提升
- 後續重構與測試不易

### 3.6 操作回饋與錯誤處理尚未完全標準化
目前未授權狀態已有統一處理，但其他成功、失敗與載入中狀態仍較依賴各頁面自行控制，容易導致：

- 成功與失敗提示不一致
- 使用者不確定是否已成功儲存
- 發生錯誤時缺少清楚引導
- 重複點擊與誤操作風險增加

## 四、優化方案建議

### 4.1 建立統一的 CRUD 操作規格
**建議內容：**
- 統一列表頁、查詢區、表格區、表單頁與詳情頁的操作模式
- 建立共用頁面模板與欄位規格
- 新增模組時可沿用既有模式，而非各自獨立開發

**預期效益：**
- 降低重複開發成本
- 提升模組一致性
- 降低新人熟悉系統的時間成本

### 4.2 優化列表查詢與結果頁體驗
**建議內容：**
- 統一查詢條件送出方式
- 增加重設條件功能
- 補齊空狀態、載入中、查詢失敗提示
- 顯示查詢結果數量與目前篩選狀態

**預期效益：**
- 提升查詢效率
- 降低操作疑惑
- 讓管理者更容易理解資料結果

### 4.3 優先重構福利政策大型表單
**建議內容：**
- 將表單分為幾個主要區塊：
  - 基本資料
  - 適用條件
  - 福利內容
  - 洽辦資訊
  - 發布設定
- 補強必填提示、錯誤訊息與欄位定位
- 評估加入草稿暫存、預設值與儲存前摘要確認

**預期效益：**
- 降低填寫負擔
- 減少漏填錯誤
- 提升高複雜度資料維護效率

### 4.4 整理權限管理機制
**建議內容：**
- 將權限邏輯改為設定化管理
- 以角色對應 route、menu、按鈕與操作行為
- 建立角色與功能權限對照表

**預期效益：**
- 降低權限邏輯維護成本
- 提升新增功能時的穩定性
- 有助於後續交接與文件整理

### 4.5 將 API 呼叫層模組化
**建議內容：**
- 依模組拆分 API service，例如：
  - news
  - articles
  - ifare
  - account
  - code
- 抽出共用的 request、auth 與 error handler
- 降低單一 API 檔案負擔

**預期效益：**
- 提升可維護性
- 模組責任更清楚
- 降低未來修改風險

### 4.6 建立一致的操作回饋機制
**建議內容：**
- 成功操作統一 toast 提示
- 失敗操作統一錯誤提示
- 表單送出中鎖定按鈕
- API 失敗時提供明確訊息或重試方向

**預期效益：**
- 提升使用者信心
- 減少誤操作
- 提升後台整體穩定感

## 五、執行優先順序建議

### 5.1 第一階段，短期優化
優先處理能快速提升操作體驗的項目：

1. 福利政策表單分區與驗證優化
2. 列表查詢體驗統一
3. 成功 / 失敗 / 載入中提示標準化
4. 空狀態與錯誤狀態補齊

### 5.2 第二階段，中期優化
著重結構與可維護性整理：

1. CRUD 共用模板化
2. 權限設定表化
3. 共用元件與操作規格統整
4. API 模組拆分

### 5.3 第三階段，長期優化
作為後台持續成長基礎：

1. 後台設計系統整理
2. 文件與程式對照補齊
3. 角色 / 功能 / API / 頁面完整對照
4. 持續觀察使用行為後再進行深度優化

## 六、預期效益

若本次後台優化方案逐步落地，預期可帶來以下效果：

### 6.1 對使用者
- 查詢與編輯流程更清楚
- 操作更穩定
- 減少出錯與重工

### 6.2 對開發與維護
- 模組結構更清楚
- 後續擴充更容易
- 共用元件與 API 維護成本下降

### 6.3 對團隊管理
- 權限與功能邏輯更容易交接
- 文件整理更有依據
- 可支持後續新功能與 PoC 延伸

## 七、結論

iFare 後台目前已具備完整功能基礎，真正需要的不是大幅重做，而是針對高頻操作、高複雜度表單與維護結構進行系統化優化。

本提案建議以**不破壞既有架構**為前提，優先聚焦在以下五個重點：

1. 福利政策大型表單優化
2. 列表查詢與操作體驗統一
3. 操作回饋機制標準化
4. 權限管理整理
5. API 與共用結構模組化

透過分階段實施，可在維持現有系統穩定的前提下，逐步提升後台效率、可用性與可維護性，並作為後續功能擴充與新功能 PoC 的良好基礎。
